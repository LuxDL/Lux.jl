59 This is only a hint for avoiding excess code generation.

63 When used in a function body, the macro must occur in statement position and

491 @boundscheck(blk)
492
493 Annotates the expression `blk` as a bounds checking block, allowing it to be elided by [`@inbounds`](@ref).
494
495 !!! note
496 The function in which `@boundscheck` is written must be inlined into
497 its caller in order for `@inbounds` to have effect.
498
499 # Examples
500 ```jldoctest; filter = r"Stacktrace:(\\n \\[[0-9]+\\].*)*"
501 julia> @inline function g(A, i)

503 return "accessing (\$A)[\$i]"
504 end;
505
506 julia> f1() = return g(1:2, -1);
507
508 julia> f2() = @inbounds return g(1:2, -1);
509
510 julia> f1()
511 ERROR: BoundsError: attempt to access 2-element UnitRange{Int64} at index [-1]
512 Stacktrace:
513 [1] throw_boundserror(::UnitRange{Int64}, ::Tuple{Int64}) at ./abstractarray.jl:455
514 [2] checkbounds at ./abstractarray.jl:420 [inlined]
515 [3] g at ./none:2 [inlined]
516 [4] f1() at ./none:1
517 [5] top-level scope
518
519 julia> f2()
520 "accessing (1:2)[-1]"
521 ```
522
523 !!! warning
524
525 The `@boundscheck` annotation allows you, as a library writer, to opt-in to
526 allowing *other code* to remove your bounds checks with [`@inbounds`](@ref).
527 As noted there, the caller must verifyusing information they can accessthat
528 their accesses are valid before using `@inbounds`. For indexing into your
529 [`AbstractArray`](@ref) subclasses, for example, this involves checking the
530 indices against its [`axes`](@ref). Therefore, `@boundscheck` annotations
531 should only be added to a [`getindex`](@ref) or [`setindex!`](@ref)
532 implementation after you are certain its behavior is correct.
533 """

542
543 In the example below the in-range check for referencing
544 element `i` of array `A` is skipped to improve performance.
545

561 that all accesses are in bounds.
562 """

567 Expr(:inbounds, :pop),

721 """
722 invoke_in_world(world, f, args...; kwargs...)
723
724 Call `f(args...; kwargs...)` in a fixed world age, `world`.
725
726 This is useful for infrastructure running in the user's Julia session which is
727 not part of the user's program. For example, things related to the REPL, editor
728 support libraries, etc. In these cases it can be useful to prevent unwanted
729 method invalidation and recompilation latency, and to prevent the user from
730 breaking supporting infrastructure by mistake.
731
732 The current world age can be queried using [`Base.get_world_counter()`](@ref)
733 and stored for later use within the lifetime of the current Julia session, or
734 when serializing and reloading the system image.
735
736 Technically, `invoke_in_world` will prevent any function called by `f` from
737 being extended by the user during their Julia session. That is, generic
738 function method tables seen by `f` (and any functions it calls) will be frozen
739 as they existed at the given `world` age. In a sense, this is like the opposite
740 of [`invokelatest`](@ref).
741
742 !!! note
743 It is not valid to store world ages obtained in precompilation for later use.
744 This is because precompilation generates a "parallel universe" where the
745 world age refers to system state unrelated to the main Julia session.
746 """

749 if isempty(kwargs)
750 return Core._call_in_world(world, f, args...)
751 end
752 return Core._call_in_world(world, Core.kwfunc(f), kwargs, f, args...)
753 end
